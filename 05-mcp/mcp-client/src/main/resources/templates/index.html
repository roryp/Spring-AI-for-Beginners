Agent Chat — Let the LLM Drive the MCP Tools

The board above uses direct MCP tool calls — the client picks which tool to invoke. This panel uses the LLM-orchestrated pattern: the same MCP tools are wired into a ChatClient, and the model decides when and how to call them. A MessageChatMemoryAdvisor remembers prior turns (notably the active gameId) so follow-up questions like "Which positions are still open?" just work — and both paths share the same game, so the board above updates whenever the agent moves a piece. Try "Start a new game and place my X in the center, then let the AI move".

How It Works — MCP Architecture

🌐

MCP Client

Two paths share one ToolCallbackProvider

  • Direct: ToolCallback.call()
  • Agent: ChatClient + tools
  • Thymeleaf frontend + REST API
  • Auto-discovers server tools
MCP Streamable HTTP
🎯

MCP Server

Game Engine + AI + @McpTool

  • startNewGame()
  • makeMove(id, pos, player)
  • aiMove(id) — LLM-powered
  • getBoardState(id)
  • getAvailableMoves(id)
1

You Click a Cell

Your move is sent to the MCP server's makeMove tool via the client's ToolCallbackProvider

2

AI Thinks

The client calls the server's aiMove tool — the server queries Microsoft Foundry via ChatClient to pick the best position

3

AI Moves

The server executes the AI's chosen move internally and returns the updated board via MCP
